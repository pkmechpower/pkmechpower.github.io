    UnitTest Class

    Overview

    UnitTest is the base test class that provides logging capabilities and utility functions for all tests in the MP test framework. It extends Google Test's ::testing::Test class and serves as the foundation for both unit tests and integration tests.

    Location

    1
    software/tests/_test_helpers/mp_unit_test.h

    Class Hierarchy

    1
2
3
4
    ::testing::Test (Google Test)
    └── UnitTest
        └── RTOS_Test
            └── MCS_Back_Test

    Key Features

    • Multi-Logger Support: Create multiple named loggers per test
    • Automatic File Management: Loggers are automatically created and cleaned up
    • CSV Output: Structured logging to CSV files for data analysis
    • Test Context Integration: Log file names automatically include test suite and test name
    • Console Output: Colored console messages for information and errors
    • Directory Management: Automatic creation of log directory structure

    Core Methods

    Logger Management

    create_logger()

    1
    bool create_logger(std::string const &name);

    Creates a named logger for the current test. The logger outputs to a CSV file with automatic naming.

    Parameters:

    • name: Unique identifier for this logger within the test

    Returns:

    • true: Logger created successfully
    • false: Logger with this name already exists

    File Naming Convention:

    1
    logs/<TestSuite>_<TestName>_<LoggerName>.csv

    Example:

    1
2
3
4
5
6
7
8
    TEST_F(MyTest, VoltageMonitoring) {
    create_logger("Voltage Data");
    create_logger("Error Status");

    // Creates:
    // logs/MyTest_VoltageMonitoring_Voltage_Data.csv
    // logs/MyTest_VoltageMonitoring_Error_Status.csv
}

    Special Character Handling: Whitespace in logger names is automatically converted to underscores:

    1
    create_logger("Motor Driver Status");  // Creates: Motor_Driver_Status.csv

    get_logger()

    1
    LoggerProxy get_logger(std::string const &name);

    Retrieves a previously created logger for writing data.

    Parameters:

    • name: Name of the logger (must match name used in create_logger())

    Returns: LoggerProxy object for stream-based logging

    Example:

    1
2
    create_logger("Data");
get_logger("Data") << "Time" << "Value" << "Status";

    Console Output

    print_info()

    1
    static void print_info(std::string_view const &message);

    Prints an informational message to the console in yellow color.

    Format:

    1
    [   INFO   ] <message>

    Example:

    1
2

UnitTest::print_info("Waiting for state transition");

    print_error()

    1
    static void print_error(std::string_view const &message);

    Prints an error message to the console in red color.

    Format:

    1
    [   ERROR   ] <message>

    Example:

    1
2
3
    if (!condition_met) {
    UnitTest::print_error("Timeout waiting for response");
}

    Test Lifecycle Hooks

    TearDown()

    1
    virtual void TearDown() override;

    Called after each test. Automatically clears all loggers and calls base class teardown.

    Important: When overriding, always call the base method:

    1
2
3
4
    void TearDown() override {
    // Your cleanup code
    Base::TearDown();  // Essential!
}

    SetUpTestSuite() / TearDownTestSuite()

    1
2
    static void SetUpTestSuite();
static void TearDownTestSuite();

    Called once before/after all tests in the test suite. Default implementation calls Google Test base methods.

    create_logger("Simulation");
    get_logger("Simulation") << "Time" << "Position" << "Velocity" << "Acceleration";

    float dt = 0.01f;
    for (int step = 0; step < 1000; step++) {
        float time = step * dt;

        // Compute values
        float pos = compute_position(time);
        float vel = compute_velocity(time);
        float acc = compute_acceleration(time);

        // Log every 10th step to reduce file size
        if (step % 10 == 0) {
            get_logger("Simulation") << time << pos << vel << acc;
        }
    }
}

    class MySystemTest : public RTOS_Test {
protected:
    void SetUp() override {
        Base::SetUp();

        create_logger("Continuous Monitor");
        get_logger("Continuous Monitor") << "Time" << "State" << "Value";

        // Register periodic logging
        register_simulation_update([this]() {
            get_logger("Continuous Monitor")
                << get_simulation_time()
                << get_current_state().c_str()
                << get_monitored_value();
        });
    }

    Directory Creation

    The logs/ directory is created automatically on first logger creation:

    1
2
    // First call in any test
create_logger("Data");  // Creates logs/ directory if it doesn't exist

    File Overwriting

    Important: Log files are overwritten on each test run. If you need to preserve logs:

    1. Copy logs directory after test run
    2. Rename test to create different files
    3. Use version control to track changes

    8. Flush Important Data

    The logger automatically flushes after each line, ensuring data is written even if the test crashes:

    1
2
3
    // Each call to get_logger() creates a LoggerProxy that
// flushes in its destructor
get_logger("Critical") << time << important_value;  // Auto-flushed

    Common Pitfalls

    1. Forgetting to Create Logger

    1
2
3
4
5
6
    // WRONG - will assert/crash
get_logger("Data") << 1.0f << 2.0f;  // Logger "Data" doesn't exist!

// CORRECT
create_logger("Data");
get_logger("Data") << 1.0f << 2.0f;

    2. Creating Logger Multiple Times

 3

 9

    3. Not Writing Headers

    1
2
3
4
5
6
7
8
    // WRONG - CSV without headers is hard to interpret

// CORRECT
create_logger("Data");
get_logger("Data") << "Time" << "Temp" << "Pressure";

    4. Incorrect Data Alignment

    1
2

get_logger("Data") << 1.0f << 2.0f;        // 2 values!

// CORRECT
get_logger("Data") << "A" << "B" << "C";
get_logger("Data") << 1.0f << 2.0f << 3.0f;

    Limitations

    1. CSV Format Only: Loggers output CSV format only
    2. No Custom Delimiters: Comma is the only delimiter
    3. No Binary Logging: All data is text-based
    4. No Log Rotation: Files are overwritten on each test run
    5. No Runtime Log Level: All created loggers are active
