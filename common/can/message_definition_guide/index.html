    Message Definition & Generation Guide

    Warning

    Outdated

    This guide explains how to add a new message to the system, including CAN and UART configuration, and generate the necessary C++ and Python code using the generator tool.

    File Locations

    Purpose Path / Folder
    Message definitions (YAML) messages.yaml
    Generator script generator.py
    Jinja2 templates templates/
    Generated output gen/

    Adding a New Message

    1. Edit messages.yaml

    Every message belongs under the top‑level messages: list. Provide at least the following keys:

    Key Required? Description
    name ✔ Descriptive class‑style name, e.g. Motor_Temperature
    id ✔ Numeric Active::Message ID (0–255, unique)
    source ✔ Node that transmits the message (must exist in nodes:)
    receivers ✔ (list) Nodes that receive the message
    payload optional Ordered list of field definitions (see below)
    transports ✔ Transport‑specific settings (CAN / UART) 
    - name: Motor_Temperature
  id: 200
  source: MOTOR_DRIVER_LEFT
  receivers: [MCS_MAIN, LOGGER_PC]
  description: Temperature of motor driver IGBT
  payload:
    - { name: igbt_temp_celsius, type: float, uart_type: float }
  transports:
    can:
      dlc: 3
      layout:
        - { byte: 1, field: igbt_temp_celsius }
    uart: {}

    2. Field Reference

    Field key Meaning
    name C++ tuple field name
    type Raw CAN storage type (uint8, uint16, float, …)
    uart_type Type when sent over UART (defaults to type)
    encode C++ function to pack raw value into CAN byte(s)
    decode C++ function to unpack raw value from CAN byte(s)
    <<: *alias Merge properties from a common anchor (see common: section)

    3. Transport Details

    CAN settings

    transports:
  can:
    dlc: 5                     # data‑length code (1‑8 bytes)
    layout:
      - { byte: 1, field: brake_pct }
      - { byte: 2, field: accel_pct }
      - { byte: 3, field: steering, slice: hi }
      - { byte: 4, field: steering, slice: lo }
    • layout.byte is 1‑based for readability.
    • Use slice: hi/lo to split 16‑bit fields.

    Disable CAN for a message:

    transports:
  can: { disabled: true }

    UART settings

    transports:
  uart: {}      # present ⇒ message is sent over UART

    If uart is omitted the message is not transmitted over UART.

    Generator Usage

    python generator.py

    The script will:

    1. Parse messages.yaml
    2. Compute derived data (CAN IDs, C++ types, decode expressions)
    3. Render templates into gen/
    4. Optionally run clang-format on C/C++ output
    Generated file Contents
    active_messages.h using Foo_Msg = Active::Message<…>
    can_traits.h CAN_Message_Traits<> encode/decode impl
    message_forwarder.h Switch logic for ETL⇄CAN forwarding
    gen/uart_debug.py Python decoder for UART stream

    Workflow Recap

    1. Add / edit messages in messages.yaml.
    2. Run python generator.py.
    3. Inspect gen/ output (code‑review encouraged).
    4. Build firmware; include generated headers.
    5. Use python gen/uart_debug.py COM5 115200 to inspect UART data.

    Troubleshooting

    Symptom Likely Cause & Fix
    TemplateSyntaxError with for Avoid list comprehensions inside {{ … }} blocks
    Unknown message ID in UART debug Message lacks transports: uart section
    CAN decode returns zeros Field missing from layout or bad byte indices
    Duplicate IDs / collisions Keep id unique; generator will warn in future
